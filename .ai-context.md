# Resid Manager - AI Context & Project Architecture

## Technical Stack
- **Architecture:** Kotlin Multiplatform (KMP)
- **Backend Module (`:backend`):** Ktor Server (JVM), Framework: JetBrains Exposed SQL, Database: PostgreSQL.
- **Frontend Module (`:frontend-web`):** Jetpack Compose for Web (Kotlin/JS - Compose HTML/Canvas).
- **Shared Module (`:shared`):** Common Kotlin code (JVM & JS targets). Contains Serializable DTOs, data models, and shared validation logic.
- **Dependency Management:** Centralized via Gradle Version Catalog (`gradle/libs.versions.toml`).

## Project Module Structure
- `:shared` -> Core business logic, domain models, and network DTOs shared between client and server.
- `:backend` -> Standalone Ktor server application. Depends on `:shared`.
- `:frontend-web` -> Declarative web UI using Jetpack Compose for Web. Depends on `:shared`.

## Core Business Rules & Domain Logic
1. **Currency:** The default pivot currency for all residences is FCFA (XOF). All financial metrics are aggregated in this currency.
2. **Electricity Statements:** Calculated automatically using the formula: `Amount Due = (New Index - Previous Index) * kWh Price`. Input validation is strict: `New Index` MUST be greater than or equal to `Previous Index`. The previous index is always read-only.
3. **Lease Workflow (Baux):** Contracts follow a strict state machine: `PENDING_PAYMENT` -> `DOWN_PAYMENT_PAID` (if partial payment) -> `PENDING_SIGNATURE` (when fully paid) -> `SIGNED_ACTIVE`.
4. **Maintenance Tickets:** Workflow: `OPEN` -> `IN_PROGRESS` -> `CLOSED`. Closing a ticket requires an `intervention_cost`. Upon closure, this amount is automatically injected into the financial engine as a residence operational expense.
5. **Automated Invoicing:** Every 1st of the month at 00:00, the system automatically generates an `UNPAID` rent transaction for every `SIGNED_ACTIVE` lease.
